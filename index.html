Reactive Apps

The new paradigm in modern web sites is the evolution to reactive applications. What exactly is a reactive application and why should we care?

Reactive applications are desktop-type apps for the web with sophisticated domain-specific behavior to “consumerize” them so that they guide the user along the right paths to get their jobs done correctly, quickly and without frustration.

They’re what most businesses are aiming for today whenever they build an internal or business-to-business application. If you’re a developer or stakeholder involved in building such an application, you know how difficult they can be to construct within the browser.

It was easier when you didn’t have to build that behavior. Old-school web applications with postbacks and full-page refreshes are not reactive applications because they don’t respond and guide the user in real-time. Users are left to wait until a page rebuilds to find out what they did wrong or what to do next. For complex business domains, the pages have to be artificially split to respond quickly and the division of united tasks can further confuse the user. So while easier on the developers and stakeholders to define and build, they are no longer enough.

Development and Testability of Reactive Apps

There are all sorts of application frameworks that aim at making JavaScript web application development easier. They can successfully deliver rich and pleasing layouts across different browser environments with minimum work, making it seem as if the application is “almost done” and just needing a little polish.

But reactive applications are not simply about eye candy–they’re about delivering business value by making the experience simple and directed for the user. They do this by continually updating the screen to instantly respond to user input and the current state of the data filtered through context-specific business rules. Here are some of the common patterns you might find in a reactive application:

  • Action buttons that enable and disable immediately based on the state of your application and what is selected. For instance, an “apply” button that is active only when all selected rows in a grid meet certain business requirements.
  • Calculated fields and status icons based on the values of other fields that immediately update when any of the source fields change.
  • Areas of a form that enable or expand immediately based on a parent checkbox being activated or a configuration across multiple fields detected.
  • Tabs that are automatically selected based on the result of actions or the type of data selected.
  • An options box that controls validation thresholds for other screens of the app. When changed, any affected screen calculations update themselves.
  • A detail panel that dynamically changes what it displays based on the type and number of rows selected. For instance, it may show a customized truck panel for a truck asset versus a computer asset, or an aggregate chart if multiple items are selected.

Capturing all of this domain-specific behavior and making it work correctly is much more difficult than composing the layout. And building it in such a way that it survives the inevitable course corrections and reorganizations along the way without becoming a complete mess is even harder. The bar has been raised. There is no going back. The challenge now is to effectively deal with the daunting complexity of specifying and building a rich reactive application.

The Right Architecture for Reactive Apps

With the emergence of javascript intensive web applications, more and more behavioral logic has migrated to the browser. This means that it becomes an imperative to be able to migrate your behavioral tests to the client code as well. If you don’t then you are building a house of cards that will not sustain the weight of additional functionality as your product matures in its feature set.

Frameworks have emerged to assist in isolating your behavioral logic such that it can be tested. The MVVM design pattern has arguably been the most successful at facilitating behavioral testing of reactive client code.

Unfortunately however, Sencha Extjs, the javascript widget library that is used by 55% of Fortune 100 companies does not furnish an MVVM pattern leaving one with a less than perfect framework for true behavioral testing.
